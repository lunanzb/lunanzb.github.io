c为什么在C中对向量使用索引运算符被认为是不好的风格堆栈内存溢出

发表于 2020-08-28 |

原文标题:

Why is it considered bad style to use the index operator on a vector in C++?

问题:

我正在使用矢量的程序。 所以我要做的第一件事就是声明我的向量。 （顺便说一句，这也被认为是不好的做法吗？我应该只输入std::vector x(10)吗？） 然后，我继续为向量分配值，并要求其大小。 我不知道它会返回0 ，所以经过一番搜索，我发现我需要使用push_back方法而不是索引运算符。 现在返回10 。 因此，我想知道的是为什么索引运算符允许我在给定索引下访问向量x中“存储”的值，但不会更改其大小。 另外，为什么这种不好的做法呢？

解决方案:

===============>>#1 票数：5

当您执行x.reserve(10) ，仅将容量设置为十个元素，但大小仍为零。 这意味着，然后在循环中使用index运算符将超出范围（因为大小为零），并且将具有未定义的行为 。 如果要设置大小，则在构造向量时可以使用resize或简单地告诉它： 至于向量的容量 ，当您设置它（使用例如reserve ）时，它将为（在您的情况下）十个元素分配所需的内存。 这意味着当您执行push_back ，将不会重新分配矢量数据。 如果您不更改容量或添加超出容量的元素，则每个push_back可能导致重新分配矢量数据。

===============>>#2 票数：3 已采纳

听起来您确实在问为什么事情是这样。 大部分归结为效率。 如果x[i]要创建一个尚不存在的值，那么效率将受到两次打击。 首先，每个索引操作都需要包括检查以查看其是否超出向量的当前大小。 其次，即使您仍要向其中分配新值，也需要默认构造新元素。 reserve和resize的原因相似。 resize要求每个元素的默认构造。 对于诸如vector类的东西来说似乎并不重要，但是对于vector ，确实确实是一件大事。 使用reserve是一种优化，完全是可选的，它使您可以预测向量的最终大小并防止其增长。 push_back避免了元素的默认构造，因为已知内容，因此它可以使用move或copy构造函数。 这些都不是坏样式，请使用适合您情况的任何样式。 **===============>>#3 票数：2** 顺便说一句，这也被认为是不好的做法吗？ 不，创建一个空向量并保留内存并不是一个坏习惯。 我应该只输入std::vector (10)吗？） 如果您打算初始化10个元素的向量，而不是清空一个元素，那么应该。 （如果您打算创建一个空向量，则否） 然后，我继续为向量分配值，并要求其大小。 这具有不确定的行为。 不要尝试访问不存在的对象。 因此，经过一些搜索，我发现我需要使用push_back方法而不是索引运算符。 那是一种选择。 另一种方法是使用构造函数初始化元素： std::vector (10) 。 还有一种方法是使用std::vector::resize 。 为什么在C ++中对向量使用索引运算符被认为是不好的风格？ 这不是一般情况。 如果索引中没有您尝试访问的元素，那么这是错误的（不仅是错误的样式）。
