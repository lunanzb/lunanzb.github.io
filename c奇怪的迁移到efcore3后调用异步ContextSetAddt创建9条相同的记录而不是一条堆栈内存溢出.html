c奇怪的迁移到efcore3后调用异步ContextSetAddt创建9条相同的记录而不是一条堆栈内存溢出

发表于 2020-08-31 |

原文标题:

Weird! After migration to ef core 3 calling async Context.Set ().Add(t) creates 9 the same records instead of one

问题:

我刚刚从Net Core 2.2迁移到了MVC Net Core 3.0。 现在，每次我尝试向数据库添加新记录时，应用程序都会在DB中创建同一记录的9个副本。 我的Db中有很多表，并且每个表上都会发生这种情况。 当我使用Net Core 2.2时，一切正常。 我正在使用以下代码向DB添加新记录： db上下文文件： 从公司数据库存储库文件调用此函数： 从公司存储库文件中调用db存储库： 从Mvc控制器调用公司存储库： 从客户端的jquery ajax调用控制器，并且对控制器的请求正在单击“添加公司”按钮上发送。 我可以在屏幕上看到“开始”警报9次。 一段时间后，我可以看到9次“成功”警报。 当我尝试在调试器中跟踪代码时，我可以看到执行从一个函数混乱地转到了另一个函数-一个函数中的一行代码几次，而另一个函数中的几次重复执行。 此后，它将返回到第一个函数的另一行，并且所有内容重复20-30次。 请帮忙。 在漫长的编程生涯中，从未见过如此怪异的东西。 最奇怪的是，所有这9条记录通常都保存在MS服务器Db中。 它们具有所有字段和键。 但是，当我在屏幕上打开它们中的任何一个时，它表明所有字段均为空。 但是，使用同一应用程序，我仍然可以正常打开并在屏幕上查看在移至Core 3.0之前保存的记录。

解决方案:

===============>>#1 票数：0

在您提供的代码中，我看不出有什么理由可以多次保存记录。 我可以看到执行从一个函数混乱地转到了另一个函数 如果有多个线程在运行，这是正常的。 调试时，Visual Studio工具栏上有一个“线程”下拉列表。 当您看到它“跳转”到代码的另一部分时，请注意在该下拉列表中选择的线程号是不同的。 但这可能表明正在处理多个请求（即您实际上正在发送9个请求）。 您应该能够在客户端（无论客户端是什么）上进行验证，或者可以删除所有断点，并在控制器操作中仅设置一个： 当该断点被击中时，按F5。 只需计算该断点被击中的次数即可。 附带一提，我不太了解您在这里做什么： 如果正确保存了savedItem.Id ，则savedItem.Id将具有一个值（假设它是由数据库生成的主键？）。 那么，为什么要搜索它是否在那里呢？ 毕竟，如果未成功保存，将引发异常。 那为什么它返回null会返回exist呢？ 请注意，出于相同的原因，这永远不会发生（如果不保存，将抛出异常，并且您永远也不会到达那行代码）。 我觉得您可以简化为： 这可以带来另一种简化（并在性能上有所提高）：如果要返回方法中唯一的await的结果，则可以取消使用async并在该方法中await ，而只返回Task 。

===============>>#2 票数：0

解决了。 我发现每个按钮“添加公司”单击触发发送多个ajax请求。 使用$（this）.off（'click'）和类似方法根本不起作用。 解： 现在一切正常。 “ stopImmediatePropagation（）”禁止所有重复的事件“ Click”触发。

===============>>#3 票数：-1
